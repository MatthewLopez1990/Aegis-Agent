- Cron scheduling, Kanban creation, voice record, meeting summary, trajectory generation/compression.

Every tool has permission, risk, schema, approval metadata, and an `implementation_status` so the UI and API can distinguish local implementations from mocks, placeholder artifacts, local facades, and backend gates. The `implemented` flag is reserved for concrete local/live tool paths; mocks, metadata-only fallbacks, placeholder artifacts, and backend gates are surfaced as limited rather than ready. High-risk tools return approval-required results unless approved through policy. Tool outputs are audited and kept behind connector/tool taint boundaries.

Subagent review-packet creation is exposed through `agents review-packet <card-id>` and `POST /subagents/review-packet`, not as an arbitrary free-form tool call. It writes sanitized private packet artifacts and checksum receipts for operator/model review handoff while excluding raw delegation instructions, worker stdout/stderr, worker result payloads, raw prompts, and secret values; creating the packet does not forward raw instruction or worker output to a model. `agents verify-packet <packet-id-or-path>` and `POST /subagents/verify-packet` verify packet checksum, schema, and no-raw-content controls from the private packet directory without echoing arbitrary packet fields.

Image generation and edit-preview tools create deterministic local PNG files from prompt and source metadata after approval. They are useful for exercising artifact, review, and GUI flows without a network image provider, but they do not claim photorealistic model output.

When a tool returns an artifact under `.aegis/tool-artifacts`, the local API adds an authenticated `artifact_url` such as `/tool-artifacts/<filename>`. Generated media artifacts and prompt-free metadata sidecars are written with private file permissions, the artifact directory is kept private, and the endpoint is constrained to a single artifact filename, rejects traversal, sends `X-Content-Type-Options: nosniff`, and only serves safe local content types for GUI preview/download workflows.

Web search live reads use the `http` connector. To enable them, configure `live_http_reads = true`, allowlist the provider domain, and pass `query` plus a provider-specific `provider_url` that returns JSON with `results`, `items`, `organic_results`, or `webPages.value`. Without `provider_url`, the tool performs a bounded local workspace text search and marks that a live connector is required for real web search.

GitHub issue and pull-request live reads also use the `http` connector. Pass `provider_url` or `api_url` for an allowlisted GitHub-compatible JSON endpoint; Aegis normalizes core fields and marks results as tainted web content. GitHub issue creation and pull-request comments are mock summaries by default, and can run as approved brokered-token live writes when `live_github_writes = true`, the API host is allowlisted, `api_url` or `provider_url` is supplied, and the requested `token_secret` is available through the secrets broker. Live GitHub write receipts return status, payload keys, byte counts, hashes, and an explicit redacted receipt schema instead of raw response bodies or secret values.

GitLab issue and merge-request live reads use the same governed HTTP path with GitLab-compatible JSON endpoints. GitLab issue creation and merge-request notes are mock summaries by default, and can run as approved brokered-token live writes when `live_gitlab_writes = true`, the API host is allowlisted, `api_url` or `provider_url` is supplied, and `GITLAB_TOKEN` or the requested `token_secret` is available through the secrets broker. Live write receipts use the same redacted receipt schema and explicitly report that raw response bodies and secret values are excluded.

Calendar and contacts live reads use the `http` connector too. Pass `provider_url` or `api_url` for an allowlisted Microsoft Graph or Google-style JSON endpoint; Aegis accepts top-level `value`, `items`, `data`, `events`, or `contacts` arrays and normalizes event/contact fields as tainted web content. Calendar event creation, contact creation/update, and Graph-compatible email draft/send are mock summaries by default, and can run as approved brokered-token HTTPS live writes when the matching `live_graph_calendar_writes`, `live_graph_contact_writes`, or `live_graph_email_writes` flag is true, the API host is allowlisted, `api_url` or `provider_url` is supplied, and the requested `token_secret` or default `GRAPH_TOKEN` is available through the secrets broker. Live write receipts use redacted summaries with hashes, byte counts, keys, and explicit `raw_secret_values_included=false` and `raw_response_body_included=false` flags.

Service-desk live reads use the `http` connector. Pass `provider_url` or `api_url` to `service_ticket_read` for an allowlisted ServiceNow/Jira-style JSON endpoint; Aegis accepts `tickets`, `result`, `value`, or `items` shapes and normalizes ticket fields as tainted web content. `service_ticket_write` supports approved mock create, update, and close actions by default, and can run as approved brokered-token HTTPS live writes when `live_service_desk_writes = true`, the API host is allowlisted, `api_url` or `provider_url` is supplied, and the requested `token_secret` or default `SERVICE_DESK_TOKEN` is available through the secrets broker. Live write receipts return status, ticket metadata, payload keys, byte counts, hashes, and the redacted receipt schema instead of raw response bodies or secret values.

Weather live reads use the same `http` connector as web extraction. To enable them, configure `live_http_reads = true` and allowlist the provider domain, then pass `location`, `latitude`, and `longitude` to the `weather` tool. Without coordinates, the tool stays in local placeholder mode.

Geocoding live reads also use the `http` connector. Pass `address` plus a provider-specific `provider_url`; Aegis parses common JSON coordinate shapes and stores the response as tainted web content. Without `provider_url`, `maps_geocode` stays in deterministic local placeholder mode.

Translation uses a deterministic local glossary for common operational terms in Spanish, French, and German. It reports coverage and `partial_glossary` quality when unknown terms are preserved instead of pretending to provide full machine translation.

TTS produces an approved local WAV tone cue by default. When `live_rest_writes = true`, the provider domain is allowlisted, the call is approved, and `AEGIS_MEDIA_PROVIDER_TOKEN` or a requested `token_secret` is configured through the secrets broker, `tts`, `image_generate`, `image_edit`, `voice_transcribe`, and `video_generate` can call a provider URL over HTTPS and persist only returned local artifacts, transcript output, or video job metadata plus provider domain/status, payload hash, and redacted receipt metadata. `image_generate` supports `provider_adapter: "openai_images"` for OpenAI-style image JSON requests and `data[].b64_json` responses; `image_edit` supports `provider_adapter: "openai_image_edit"` for OpenAI-style multipart edit requests with a workspace-scoped PNG, JPEG, or WEBP source image; `tts` supports `provider_adapter: "openai_tts"` for OpenAI-style speech JSON requests and binary audio responses; `voice_transcribe` supports `provider_adapter: "openai_transcription"` for OpenAI-style multipart uploads of workspace-scoped FLAC, MP3, MP4, MPEG, MPGA, M4A, OGG, WAV, or WEBM files; `video_generate` supports `provider_adapter: "openai_video"` for OpenAI-style submit, status, download, and delete actions. Raw prompt/text, source image bytes, source audio bytes, raw video response bodies, and raw secret values are not stored in receipts.

Voice recording creates a local WAV silence artifact with the requested duration for testing capture, upload, and transcription flows. It does not claim microphone access; real microphone capture remains a gateway-specific integration.
