# Feature Parity Plan

This document maps the public Hermes Agent and OpenClaw feature surface into Aegis Agent capabilities while preserving Aegis security controls. The goal is support parity through safe interfaces first, then selectively enable live integrations behind permissions, approvals, taint tracking, and audit receipts.

## Current Public Feature Surface

Evidence used for the May 2026 pass:

- Hermes README describes a self-improving agent with model switching, full TUI, messaging gateway, closed learning loop, scheduler, subagents, multiple terminal backends, and research tooling.
- Hermes v0.13 release notes emphasize durable Kanban, checkpoints, gateway auto-resume, watchdog scheduling, default redaction, role allowlists, stranger rejection, and OAuth TOCTOU fixes.
- OpenClaw docs describe a single Gateway, built-in and plugin channels, multi-agent routing with isolated sessions, streaming and chunking, 35+ providers, media, Web Control UI, mobile nodes, browser automation, web search, cron jobs, skills, plugins, and workflow pipelines.
- OpenClaw TUI docs describe local and gateway mode, header/status/footer, chat log with tool cards, autocomplete, slash commands, local shell commands, history, streaming, and connection repair.

## Implemented Support

### Memory and Self-Improvement

- Governed memory records with provenance, confidence, sensitivity, confirmation, deletion, conflict detection, duplicate merging, audited conflict resolution strategies, unresolved-conflict surfacing in live model context, trusted-session memory candidate previews and explicit commits that skip untrusted turns, block secret-like content, and preserve session/message provenance, review queues, review digests, overdue review escalation summaries with per-route team policies, single-record and batch confirm/delete actions for uncertain or stale records, scheduled operator digest and overdue-escalation delivery through approval-gated channels, dry-run recertification previews plus configurable per-type recertification marking for old confirmations, configurable retention TTLs, expiration, scoped lexical plus local semantic retrieval, and audit.
- Memory health reports provide an enterprise readiness snapshot: local provenance-aware quality scoring, duplicate and conflict recommendations, low-confidence and stale-confirmation review signals, aggregate-only audit receipts, dashboard `memory_readiness`, and CLI/TUI `memory health` commands.
- Failed tasks produce durable self-repair proposals with structured failure class, target subsystem, proposed action, and required validation metadata. Reviewed proposals can generate isolated no-mutation repair plan sandboxes, emit redacted model-ready synthesis prompt packets with strict JSON schema and workspace-mutation constraints, synthesize model-style patch candidates into workspace-scoped unified diffs with sandbox verification receipts, or collect pending repair candidates with planned files, patch plans, and unified diffs that must pass `git apply --check` before approval. Syntheses may echo a `prompt_id`, and Aegis verifies the private prompt artifact belongs to the same proposal and still matches its checksum sidecar before attaching lineage and the prompt artifact SHA-256 to the candidate. Candidate review is separate from proposal approval: CLI/API/TUI/GUI controls must approve a patch candidate before it can mutate the workspace, and application rechecks linked prompt lineage before any patch touches files. Approved candidates can apply those diffs through repair controls, roll back an applied candidate before verification, and remain `applied_pending_verification` until an allowlisted verification command exits successfully. Passing verification marks the candidate `verified` with the verification receipt attached, and candidate-linked verification is rejected unless that exact candidate is in the applied-pending-verification state. Implemented repair attempts require changed-file evidence plus a passed verification command; Git workspaces also require the claimed files to have visible changes. Task evidence bundles include linked repair proposals, repair candidates, repair attempts, verification receipts, learned procedural memories, missing-evidence diagnostics, and repair readiness summaries that flag missing candidates, pending reviews, unverified applied patches, and missing procedural memories. Verification receipts capture redacted output, verified repairs are promoted into procedural memory, and future repair proposals attach related self-repair procedural memories to redacted synthesis prompts so repeated failures can reuse prior lessons.
- Skill manifests with signed package metadata, permissions, risk classification, tests, evals, rollback, sandbox profile, static safety scans, enable/disable controls, runtime permission enforcement, runtime sandbox-profile enforcement, and no-network plus ephemeral Python subprocess adapters for generated skills with timeout, output-limit, and OS resource-limit receipts.
- Workflow candidate skill, skill template generation, and curator draft candidates keep generated procedures disabled or approval-required until reviewed; draft candidates store only observed-task hashes/counts before approved disabled installation.

## Safety Delta Over Hermes/OpenClaw Defaults

- External content never becomes instructions; connector, channel, tool, and skill outputs are tainted.
- Receipts omit raw untrusted blobs and retain sanitized summaries.
- Secrets use brokered handles and are redacted from logs.
- High-risk tools, connector writes, message sends, shell execution, MCP calls, generated skills, and curator skill draft installs require approval.
- Mock adapters are the default for high-blast-radius capabilities until a scoped live connector is explicitly configured; live webhook, chat webhook, SMTP email, and messaging live-send adapters are allowlisted, brokered-secret aware, and approval-gated.

## Remaining Live-Integration Work

Aegis now supports the feature categories through secure interfaces, but most third-party services are intentionally mock/stub mode. Live parity requires per-provider credential flows, provider-specific webhook validation, rate limiting, sandbox hardening, rollback logic, and tests before enabling each adapter.
