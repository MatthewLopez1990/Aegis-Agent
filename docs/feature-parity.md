# Feature Parity Plan

This document maps the public Hermes Agent and OpenClaw feature surface into Aegis Agent capabilities while preserving Aegis security controls. The goal is support parity through safe interfaces first, then selectively enable live integrations behind permissions, approvals, taint tracking, and audit receipts.

## Current Public Feature Surface

Evidence used for the May 2026 pass:

- Hermes README describes a self-improving agent with model switching, full TUI, messaging gateway, closed learning loop, scheduler, subagents, multiple terminal backends, and research tooling.
- Hermes v0.13 release notes emphasize durable Kanban, checkpoints, gateway auto-resume, watchdog scheduling, default redaction, role allowlists, stranger rejection, and OAuth TOCTOU fixes.
- OpenClaw docs describe a single Gateway, built-in and plugin channels, multi-agent routing with isolated sessions, streaming and chunking, 35+ providers, media, Web Control UI, mobile nodes, browser automation, web search, cron jobs, skills, plugins, and workflow pipelines.
- OpenClaw TUI docs describe local and gateway mode, header/status/footer, chat log with tool cards, autocomplete, slash commands, local shell commands, history, streaming, and connection repair.

## Implemented Support

### Memory and Self-Improvement

- Governed memory records with provenance, confidence, sensitivity, confirmation, deletion, conflict detection, duplicate merging, audited conflict resolution strategies, unresolved-conflict surfacing in live model context, trusted-session memory candidate previews and explicit commits that skip untrusted turns, block secret-like content, and preserve session/message provenance, review queues, review digests, overdue review escalation summaries with per-route team policies, single-record and batch confirm/delete actions for uncertain or stale records, scheduled operator digest and overdue-escalation delivery through approval-gated channels, dry-run recertification previews plus configurable per-type recertification marking for old confirmations, configurable retention TTLs, expiration, scoped lexical plus local semantic retrieval, and audit.
- Memory health reports provide an enterprise readiness snapshot: local provenance-aware quality scoring, duplicate and conflict recommendations, low-confidence and stale-confirmation review signals, aggregate-only audit receipts, dashboard `memory_readiness`, and CLI/TUI `memory health` commands.
- Failed tasks produce durable self-repair proposals with structured failure class, target subsystem, proposed action, and required validation metadata. Reviewed proposals can generate isolated no-mutation repair plan sandboxes, emit redacted model-ready synthesis prompt packets with strict JSON schema and workspace-mutation constraints, synthesize model-style patch candidates into workspace-scoped unified diffs with sandbox verification receipts, or collect pending repair candidates with planned files, patch plans, and unified diffs that must pass `git apply --check` before approval. Syntheses may echo a `prompt_id`, and Aegis verifies the private prompt artifact belongs to the same proposal and still matches its checksum sidecar before attaching lineage and the prompt artifact SHA-256 to the candidate. Candidate review is separate from proposal approval: CLI/API/TUI/GUI controls must approve a patch candidate before it can mutate the workspace, and application rechecks linked prompt lineage before any patch touches files. Approved candidates can apply those diffs through repair controls, roll back an applied candidate before verification, and remain `applied_pending_verification` until an allowlisted verification command exits successfully. Passing verification marks the candidate `verified` with the verification receipt attached, and candidate-linked verification is rejected unless that exact candidate is in the applied-pending-verification state. Implemented repair attempts require changed-file evidence plus a passed verification command; Git workspaces also require the claimed files to have visible changes. Task evidence bundles include linked repair proposals, repair candidates, repair attempts, verification receipts, learned procedural memories, missing-evidence diagnostics, and repair readiness summaries that flag missing candidates, pending reviews, unverified applied patches, and missing procedural memories. Verification receipts capture redacted output, verified repairs are promoted into procedural memory, and future repair proposals attach related self-repair procedural memories to redacted synthesis prompts so repeated failures can reuse prior lessons.
- Skill manifests with signed package metadata, permissions, risk classification, tests, evals, rollback, sandbox profile, static safety scans, enable/disable controls, runtime permission enforcement, runtime sandbox-profile enforcement, and no-network plus ephemeral Python subprocess adapters for generated skills with timeout, output-limit, and OS resource-limit receipts.
- Workflow candidate skill and skill template generation keep generated procedures disabled or approval-required until reviewed.

### TUI and Web GUI

- `aegis tui` provides a terminal command deck with task submission, status, resume, cancellation, recent tasks, approvals, security posture, capability groups, connectors, channels, models, tools, skills, schedules, scheduled evaluation runs, sessions, work boards, backends, audit commands, slash aliases, tab completion for common command branches, and private local command history.
- `aegis serve` provides a browser GUI with task submission, pause/resume/cancel controls, plus panels for runtime health, security controls, competitive parity targets, connectors, channels, models, MCP calls, tools, schedules, scheduled evaluation runs, sessions, work boards, approvals, recent tasks, and audit logs. Operators can inspect another session's task history from the session list without switching the active transcript used for new work.
- The web GUI can stream task run events over an authenticated local `text/event-stream` endpoint, giving OpenClaw-style progressive run visibility without exposing the local API token to native `EventSource` query strings. A bounded follow mode emits task status and heartbeat frames for in-progress runs, clients can resume incrementally with `since` or `Last-Event-ID` cursors, and the GUI uses `live=1` cursor reconnects for long-running nonterminal executions. Run-event snapshots include aggregate progress metrics, sanitized provider/tool substep telemetry, and per-step progress groups above the flat event log, and SSE status/heartbeat/done frames carry the same progress block. The GUI renders status-only stream frames as visible progress cards before event rows exist. Non-terminal tasks can be paused or cancelled from CLI/API/TUI/GUI with audit receipts and session transcript continuity.
- Task evidence, timelines, and run events query the audit log by task id, so older task events remain available after unrelated runtime activity.
- Audit logs can be exported as redacted normalized SIEM JSONL with hash-chain status, event hashes, task ids, correlation ids, and event-type/task filters across CLI/API/TUI/GUI surfaces.
- Policy posture includes immutable secret-deny controls, admin approval gates, dry policy evaluation, and built-in bundles that can be exported, validated as imports, diffed, applied, scheduled as approved rollouts, activated when due by the local maintenance worker, promoted between environments, listed as promotion receipts, and rolled back for strict local, approval-first, and developer-local environments. Promotion can require clean evaluation evidence and can explicitly block on open live parity backlog items before moving policies toward production; named live-gap deferrals require an explicit operator reason and are recorded on queryable promotion receipts.
- Session-bound tasks keep their originating transcript visible across status, evidence, timeline, run-event, approval, and resume-result surfaces in the CLI, TUI, and web GUI. Resume audit events include redacted session title/channel labels for requested, task, and resolved contexts; task-linked and approval-linked session messages expose API `action_hints`; web transcript messages with task metadata expose status/events/timeline/approval actions; TUI session history shows matching command hints; TUI resume switches back to the originating conversation when the active session differs; web approval detail and transcript actions preserve operator actor, reason, and admin-decision metadata; the web approval panel keeps a bounded recent approved/denied decision history with review links; dashboard data exposes a dedicated `recent_session_tasks` feed that does not disappear when unscoped background tasks fill the global recent-task slice; and session lists include safe message counts, task counts, waiting-approval counts, and latest-task status so operators can find the original work context without exposing raw request text.
- `aegis dashboard` and `GET /dashboard` expose live runtime counts, security controls, capability groups, parity targets, implementation-readiness buckets, and a structured live-gap backlog that separates ready local/live paths from local facades, mock/placeholders, and backend-gated adapters without exposing raw secrets or payloads. Each backlog item includes required safety controls, verification gates, next steps, affected platforms, sample tool surfaces, already-live read surfaces, redacted implemented-live-adapter evidence, redacted available-live-adapter evidence, redacted implemented backend-adapter evidence, redacted available backend-adapter evidence, and browser/media hardening evidence versus remaining depth work so Hermes/OpenClaw parity work is reviewable instead of hand-waved. The browser/media gap includes an operator checklist for boundary receipts, taint preservation, artifact hashing, approval, secret-capture boundaries, media worker sandboxing, live browser automation status, provider media depth, and platform media sandbox profiles. The provider/channel live connector gap also includes an operator checklist for credential handles, network allowlists, explicit live-enable flags, human approval, receipt redaction, mock fallback, read-surface inventory, and promotion scope before real writes are enabled. The remote-backend gap includes a matching checklist for explicit backend enablement, brokered backend auth, scope limits, resource limits, rollback/cleanup receipts, disabled-backend denial, and provider lifecycle depth. The TUI dashboard and capabilities command render the same live-gap backlog plus browser/media, live connector, and remote backend readiness checklists, the web readiness cards include representative tools for each bucket, the web parity cards render both the security delta and the remaining live-integration gap for each competitor target, and the web live-gap backlog lists the concrete remaining work areas with controls and gates. - Signed webhook gateway, chat webhooks, and SMTP email are available as opt-in live channel adapters. - Inbound content is normalized and processed as untrusted chat content through the context firewall.
- Outbound channel rendering is marked pending approval by default.

### Scheduling and Work Orchestration

- Schedule manager creates paused schedules with cron-like metadata, review activation, due detection, governed run-due execution, next-run advancement, scheduled memory review digests/escalations, scheduled local evaluation reports with operator digests, and audit records.
- Lifecycle hooks can be registered from CLI, TUI, or local API for manual and task lifecycle events. Hooks use allowlisted argv execution only, default to approval-required, run with timeout and output caps from the configured workspace, avoid inherited secret env, store private hook state, and emit redacted audit receipts without blocking task execution when hook dispatch fails.
- Local plugin manifests can be installed, listed, enabled, disabled, removed, and reloaded from CLI, TUI, or token-protected local API. A plugin can own skill manifests, MCP server definitions, and lifecycle hooks, but each resource is still registered through its existing governed registry with signature/static-scan checks for skills, approval and executable allowlists for MCP/hooks, private local state, redacted audit receipts, duplicate-resource rejection, path traversal rejection, and best-effort rollback if a later resource fails during install. Marketplace discovery and update checks are metadata-only from the built-in catalog or an operator-supplied local catalog JSON; they surface install mode, manifest URL, manifest digest, update availability, blocked operations, and next review actions without downloading remote plugin bundles, importing plugin Python code, capturing marketplace tokens, auto-updating unsigned resources, or trusting a marketplace signing chain yet.
- Kanban manager provides durable boards and cards for multi-agent-style work coordination. Approved `subagent_delegate` tool calls create isolated delegation cards with role, parent task, tainted-instruction metadata, approval gate metadata, and safe status previews exposed through CLI, TUI, API, and GUI; recursive autonomous worker execution remains a tracked runtime-depth gap.
- Sessions and message history are persisted and compactable. The GUI can append non-submitting session context with an explicit trust label so imported chat, web, document, or tool output is retained as context without becoming a task request.
- Task resume defaults back to the original session when possible, records requested/result/rejected audit receipts, appends resume outcomes to the session transcript with task and approval metadata, and returns interactive TUI/web operators to the originating conversation after cross-session resume actions.
- Seven execution backends are represented as policy-visible capabilities: local, Docker, SSH, Singularity, Modal, Daytona, and Vercel Sandbox. ### MCP and Migration

- MCP server registry stores disabled-by-default server definitions with allowed tool lists and approval requirements, and can run approved one-shot stdio tool calls with executable allowlists and tainted outputs.
- OpenClaw/Hermes migration inspection is dry-run by default and blocks direct secret import. Memory preview commands parse supported memory/session files into sanitized review-required candidate records without persistence, classify likely memory types, preserve untrusted provenance, and block secret-like content. Separate memory commit commands persist only unblocked reviewed candidates through the governed memory manager while preserving platform/path/entry provenance and reviewer metadata. Local SQLite schema migrations expose checksum-verified status, dry-run plans, private SQLite backups, no-network PostgreSQL/MySQL schema plans, and private operator-runner bundles with checksum-stamped SQL files for external staging reviews.

## Safety Delta Over Hermes/OpenClaw Defaults

- External content never becomes instructions; connector, channel, tool, and skill outputs are tainted.
- Receipts omit raw untrusted blobs and retain sanitized summaries.
- Secrets use brokered handles and are redacted from logs.
- High-risk tools, connector writes, message sends, shell execution, MCP calls, and generated skills require approval.
- Mock adapters are the default for high-blast-radius capabilities until a scoped live connector is explicitly configured; live webhook, chat webhook, SMTP email, and messaging live-send adapters are allowlisted, brokered-secret aware, and approval-gated.

## Remaining Live-Integration Work

Aegis now supports the feature categories through secure interfaces, but most third-party services are intentionally mock/stub mode. Live parity requires per-provider credential flows, provider-specific webhook validation, rate limiting, sandbox hardening, rollback logic, and tests before enabling each adapter.
